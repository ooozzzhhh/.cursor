# PRD 撰写规范

按公司**产品需求文档 PRD** 格式与写作规范撰写，充分描述需求，使**开发**和**评审**能快速理解。

**文档定位**：对已有产品新增需求的实现，用于已有产品的升级完善。

**核心目标**：需求讲清楚、条件写完整、开发可落地、评审可通过。

---

## 1. 总体原则

- **读者**：主要面向**非技术背景读者**，测试、实施、评审领导等。开发（前端/后端/算法）为辅。
- **语言**：全文简体中文；术语一致，同一事物全文同一称呼；尽量用业务语言，少用实现细节术语；遇到晦涩难理解的技术点，可适当用比喻、例子解释。
- **结构**：多用**标题 + 编号列表/要点列表**，避免整段大段文字。
- **详略**：业务规则、数据逻辑、校验条件写到开发能直接实现；条件分支写全；表/接口有则给字段级说明或 DDL；枚举列出代码与含义。
- **格式**：不需要总标题，因为云文档自带标题，直接从文档说明或修订记录开始写即可；各级标题与其章节内容之间不需要空行，尽量减少空行的使用。

### 1.1 原型与 UI 类需求的写作要求（重要）

当需求对象是**原型图、UI 设计稿或前端演示项目**（如「产品原型图」下的文件）时，PRD 应以**业务逻辑**和**用户交互**为核心：

- **以业务和用户为视角**：写清楚「用户是谁、在什么场景下、要完成什么任务、看到什么、能做什么操作、系统如何反馈」，而不是「某个配置对象、某个变量、某个接口字段」。
- **避免依赖具体代码或技术实现**：不出现原型/演示项目中的变量名、配置名、类型名（如 `MODULE_CONFIGS`、`RunRecord`、`functionId`、`status` 等）；不描述「前端用哪种技术栈、如何配置路由」。产品正式实现可能在 **UHA/scp-foundation** 等仓库中采用不同技术架构（如 Vue、微前端），PRD 应保持与实现解耦。
- **输入/输出用业务语言**：输入可以写「用户要选择/填写哪些业务信息」「依赖哪些业务数据」「建议的表设计」等；输出可以写「用户会看到哪些统计、能导出哪些文件、列表展示哪些列、会连接哪些下游模块」等。具体内容可以根据需求的特点灵活判断。
- **业务流程与业务规则**：用「用户点击…」「系统展示…」「若…则…」描述操作顺序与条件；状态、按钮、筛选项等用**业务含义**（如「任务状态：成功 / 失败 / 运行中 / 已终止」），不写代码枚举值。
- **原型图小节**：重点描述界面结构、主要区域、操作与反馈。

整体行文建议采用「**总述 1～2 行 + 若干分点**」的方式，避免大段连续文字堆砌，这样既方便领导快速扫读，也便于开发根据要点实现。

## 2. 文档整体结构

| 顺序 | 内容 |
|------|------|
| 0 | 文档说明（可选） |
| 1 | 第 1 章：需求概述 |
| 2 | 第 2 章：需求范围和流程概览 |
| 3 | 第 3 章：对标分析 |
| 4 | 第 4 章：详细解决方案 |
| 5 | 第 5 章：未来扩展说明 |

### 2.0 文档说明（可选）

- 说明本文档用途：对已有产品新增需求的实现、升级完善。

### 2.1 第 1 章：需求概述

```markdown
# 1. **需求概述**

## 1.1 需求涉及的模块
[填写需求所涉及的一级模块，若涉及跨模块的场景，则填写多个一级模块。一级模块含IPS、MPS、MRP、SCH、S&OP、DFP]

## 1.2 需求名称
[将需求名称填写过来]

## 1.3 需求简述
[主要分为现状、问题、需求价值三块内容；可适当展开说明]
```

### 2.2 第 2 章：需求范围和流程概览

| 子节 | 标题 | 用途 |
|------|------|------|
| 2.1 | **基本概念说明** | 定义一级模块、二级模块；列出本需求涉及新增、修改或删除的二级模块 |
| 2.2 | **业务流程概览** | 基于主设计文档的二级流程图更新；新增/修改/删除的二级模块对流程的影响；新增模块展开三级流程图；修改模块说明与原三级流程的差异 |
| 2.3 | **数据流程概览** | 新增/修改二级模块时：列出输入数据来源、输出数据去向；删除模块时：列出剩余模块的数据流 |

### 2.3 第 3 章：对标分析——**重点**

- 说明该需求在竞品中的实现逻辑及方式。
- **至少一个竞品分析**：Blue Yonder、o9 solutions、SAP IBP 或 APO等。

### 2.4 第 4 章：详细解决方案——**重点**

- **是否拆子模块**：基于 2.2 的三级流程图，对新增的二级模块展开每个三级模块的说明；对已有的二级模块，仅对新增、修改的三级模块进行说明。
- 子模块命名能看出职责（如三级模块 1、三级模块 2）。
- 每个 4.x 下**统一包含以下 7 个子节**（无内容时写"无"或"不涉及"）：
| 子节 | 标题 | 用途 |
|------|------|------|
| 4.x.1 | **模块描述** | 本三级模块职责；若原有则给主设计文档链接；若新增或原描述不适用则给新描述 |
| 4.x.2 | **输入** | 依赖的数据、配置、接口、表结构等；若原有则标识变化 |
| 4.x.3 | **输出** | 产出的数据、文件、接口、页面等；若原有则标识变化 |
| 4.x.4 | **业务流程** | 四级流程图；若原有则标识对原流程的调整 |
| 4.x.5 | **业务规则** | 若原有则仅列新增或调整的逻辑；若新增则列全部规则 |
| 4.x.6 | **原型图** | 界面与交互说明；新增界面画新样式；已有界面调整则标明差异点 |
| 4.x.7 | **用例** | 若原有则仅列变化的用例并标识修改/删除；若新增则列全部用例 |

### 2.5 第 5 章：未来扩展说明

- 说明从整体设计考虑包含、但考虑开发优先级先不实现的内容或遗留项。
- 例如：手工排程未来拓展针对换型优化目标；优化算法本次暂不开发，在后续开发计划中实现。

## 3. 详细解决方案中，各子模块写作要点

### 3.1 模块描述（4.x.1）

- 若本三级模块原先就有，则给出产品主设计文档的链接。
- 若为新增模块，或原有模块描述已不适用，则给出新的模块描述：本模块做什么、解决什么问题、谁在用、结果给谁用。

### 3.2 输入（4.x.2）

- 输入的定义：表、字段、参数、文件。
- 若本三级模块原先就有，则标识输入的变化。
- 新增/修改表：给建表 DDL 或字段清单，关键字段中文注释与约束。
- 已有表调整：列出新增/修改字段、含义、必填、取值范围。
- 依赖其他模块或全局配置时写清来源与取值方式。

### 3.3 输出（4.x.3）

- 明确对外产出：生成/更新的表、导出文件、页面。
- 若本三级模块原先就有，则标识输出的变化。
- 数据类输出：主要字段、单位、精度；被谁使用可简要注明。
- 无对外产出时写"无"，并确认无下游依赖。

### 3.4 业务流程（4.x.4）

- 给出当前三级模块展开的四级流程图；若无则可跳过。
- 若为已有模块，产品主文档已有四级流程图，请标识对原流程作了怎样的调整。
- 步骤列表描述主流程；复杂时可拆子流程。
- 每步写清：谁/在什么入口/做什么操作，产生什么状态或数据变化。
- 分支可在此简述，详细条件放在**业务规则**。

### 3.5 业务规则（4.x.5）——**重点**

开发与测试的直接依据，**规则和条件必须清晰、可执行**。

- **若本三级模块原先就有**：仅需列出需新增或调整的逻辑，例如：
  - 增加、删除、调整列的数据项及对应逻辑
  - 新增、修改、删除的规则表及对应逻辑
  - 其它新增、调整的逻辑

- **条件表述**
  - 用"**若 … 则 …**"或"**当 … 时，…**"。
  - 条件**穷尽**：所有分支都写到，必要时补"其他情况""未配置时默认…"。
  - 关键条件用**粗体**或 `代码`（参数名、状态值、字段名）。

- **建议包含的规则类型**（按需选取）：
  - 数据来源与生成、校验规则（必填/格式/范围/唯一性/跨表一致性；触发条件与处理方式）、状态与权限、计算与统计（公式、口径、精度、单位）、与算法/引擎的约定。

- **枚举与码值**：列出代码值 + 中文含义，说明使用场景（如"仅当类型=X 时显示"）。

- **错误与异常**：校验失败、异常数据、超时等——触发条件、提示内容、是否阻断、是否可导出明细。

### 3.6 原型图（4.x.6）

- 若不需对已有界面作调整，则不必在此列出。
- 枚举该模块所有的 UI 原型图，并配上说明 UI 中各个组件的内容；原型图可使用悠桦林前端组件、Axure 等方式生成，目的是让用户直观理解、让开发能依据原型图开发。
- 特殊交互（行内编辑、批量操作、展开折叠、多选约束等）要单独说明。原型中的必填、禁用、联动需在 4.x.5 有对应规则。

### 3.7 用例（4.x.7）

- 若本三级模块原先就有，则仅需列出变化的用例，并标识原有哪些用例被修改、删除。
- 若本三级模块为全新的模块，则请列出该模块涉及的全部用例。
- 配合原型图共同使用，枚举所有需要说明的用例。
- 每条含：**前置条件**（可选）、**操作步骤**、**预期结果**。
- 覆盖主流程、主要分支、校验、异常与边界；未细化可写"待补充"。

## 4. 格式约定

- **标题**：一级 `# 1. **需求概述**`、`# 2. **需求范围和流程概览**`、`# 3. 对标分析`、`# 4. **详细解决方案**`、`# 5. **未来扩展说明**`（数字+粗体）；下级用 ##、###、#### 层级清晰。
- **列表与强调**：复杂逻辑用有序/无序列表；关键**条件、参数、状态、字段名**用粗体或反引号。枚举写"代码值 + 中文含义"。
- **数据与表**：表名、字段名与库表一致；建表/字段说明用 SQL 代码块，注释中文。数值类约定单位、小数位、取数来源。

---

## 5. Agent 使用步骤

1. **确认范围**：根据用户描述确定需求涉及的模块、需求名称、需求简述；**判断是否有主设计文档及对应关系**；**判断本次涉及新增/修改/删除的二级模块**；**若需求对象为原型/UI 设计，则遵循 1.1 以业务逻辑与用户交互为主，不依赖原型代码与实现**。
2. **搭骨架**：生成文档说明（可选）、第 1 章（1.1 需求涉及的模块、1.2 需求名称、1.3 需求简述）、第 2 章（2.1 基本概念说明、2.2 业务流程概览、2.3 数据流程概览）、第 3 章对标分析、第 4 章（按三级模块拆 4.1、4.2…，每个 4.x 下含 4.x.1～4.x.7）、第 5 章未来扩展说明。
3. **填内容**：先需求概述，再范围与流程概览，再对标分析，再按三级模块填输入、输出、流程、**业务规则**、原型图、用例；规则部分优先保证条件完整、表述无歧义；**原型/UI 类需求中避免写入变量名、配置名、技术栈，改用业务用语**。
4. **自检**：章节编号连续、术语统一、关键逻辑在业务规则中均有对应、开发与领导能据此理解和实现。
